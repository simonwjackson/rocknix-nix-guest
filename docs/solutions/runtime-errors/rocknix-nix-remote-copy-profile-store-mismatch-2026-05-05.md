# ROCKNIX nix-portable remote copy can create profile/store visibility mismatches

## Context

While deploying a Korri Electrobun desktop build to the AYN Odin 2 Portal (`root@thor`), we built the package on another aarch64 host (`yari`) and copied it to ROCKNIX.

Build host:

```sh
ssh yari 'cd /tmp/korri-build && nix build .#korri-desktop-odin --no-link --print-out-paths'
# /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
```

Target host:

```sh
ssh root@thor 'command -v nix || true; command -v nix-store || true'
# non-login SSH PATH was /usr/bin:/bin:/usr/sbin:/sbin
# /storage/bin/nix existed, but was not on PATH
# no nix-store wrapper existed initially
```

`/storage/bin/nix` on `thor` was a nix-portable wrapper:

```sh
#!/bin/sh
# Generated by nix-portable-install. Re-run installer to repair.
exec "/storage/apps/nix-portable/bin/nix-portable-run" nix "$@"
```

## What went wrong

### 1. `ssh-ng` plus nix-portable daemon did not materialize the store path in the host-visible `/nix/store`

A temporary daemon wrapper was created:

```sh
cat >/storage/bin/nix-daemon <<'EOF'
#!/bin/sh
exec /storage/bin/nix daemon "$@"
EOF
chmod +x /storage/bin/nix-daemon
```

Then the closure was copied from `yari`:

```sh
nix copy --no-check-sigs \
  --from ssh://yari \
  --to 'ssh-ng://root@thor?remote-program=/storage/bin/nix-daemon' \
  /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
```

The copy command completed, and `/storage/bin/nix path-info` on `thor` could see the path:

```sh
/storage/bin/nix path-info /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
# /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
```

But normal filesystem access could not:

```sh
ls -ld /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
# ls: /nix/store/...: No such file or directory
```

This strongly suggests that the nix-portable daemon path was operating inside a different mount/store namespace than normal host filesystem access, or otherwise recording Nix metadata without placing objects in the real mounted `/nix/store` used by the rest of the system.

### 2. Profile operations through `/storage/bin/nix` became inconsistent

After the `ssh-ng` copy, profile operations using the nix-portable wrapper briefly produced bad state:

```sh
/storage/bin/nix profile install --profile /storage/.nix-profile \
  /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
```

Observed symptoms:

```sh
/storage/bin/nix profile list --profile /storage/.nix-profile
# Name: korri-desktop
# Store paths: /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
#
# Name: korri-desktop-1
# Store paths: /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0

ls -la /storage/.nix-profile/bin
# ls: /storage/.nix-profile/bin: No such file or directory
```

The profile metadata referenced the desired package, but the profile symlink target was not usable as a normal profile with `bin/korri-desktop-odin`.

### 3. Signature trust blocked direct remote copies until explicitly disabled

The first copy attempt failed with:

```text
error: cannot add path '/nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0' because it lacks a signature by a trusted key
```

For this one-off direct copy from a trusted personal aarch64 builder, `--no-check-sigs` was used.

## Working workaround

Use the real Nix tools already present in the persistent `/nix/store`, not the nix-portable wrapper, for remote store protocol and profile installation.

A real `nix-store` existed inside an existing profile closure:

```sh
/nix/store/jw1c9x0gnqjybxkqikhapwc194kwr701-profile/bin/nix-store --version
# nix-store (Nix) 2.34.7
```

Create a wrapper for SSH store access:

```sh
cat >/storage/bin/nix-store <<'EOF'
#!/bin/sh
exec /nix/store/jw1c9x0gnqjybxkqikhapwc194kwr701-profile/bin/nix-store "$@"
EOF
chmod +x /storage/bin/nix-store
```

Copy using the classic `ssh://` store and the real `nix-store` remote program:

```sh
nix copy --no-check-sigs \
  --from ssh://yari \
  --to 'ssh://root@thor?remote-program=/storage/bin/nix-store' \
  /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
```

This materialized the path in the host-visible store:

```sh
ls -ld /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
# dr-xr-xr-x ... /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0

find /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0 -maxdepth 2 -type f -o -type l | head
# /nix/store/.../bin/korri-desktop
# /nix/store/.../bin/korri-desktop-odin
```

Then install the profile using the real `nix`, not `/storage/bin/nix`:

```sh
NIX=/nix/store/jw1c9x0gnqjybxkqikhapwc194kwr701-profile/bin/nix
$NIX profile install --profile /storage/.nix-profile \
  /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0
```

Final verification:

```sh
readlink -f /storage/.nix-profile/bin/korri-desktop-odin
# /nix/store/v9zvh5zv70prgh3xjnnmb6ssgrnjmyip-korri-desktop-1.0.0/bin/korri-desktop-odin
```

## Open questions for follow-up

1. Why does `ssh-ng://root@thor?remote-program=/storage/bin/nix-daemon` report a successful copy while the path is not visible at `/nix/store/...` from the host shell?
2. Is nix-portable still expected to be used after Layer 4 real Nix is installed, or should `/storage/bin/nix` be replaced with a real-Nix shim?
3. Should ROCKNIX provide `/storage/bin/nix-store` and `/storage/bin/nix-daemon` wrappers backed by real Nix once Layer 4 is installed?
4. Should non-login SSH sessions include `/storage/bin`, or should deployment docs always use absolute paths?
5. Why did `nix profile install` through the nix-portable wrapper create a profile without a visible `bin/` output, while real Nix installed the same path correctly?

## Follow-up fix

The image-first nix-integration tooling no longer ships nix-portable or standalone bootstrap tooling. `nixctl install` removes legacy portable wrappers named `/storage/bin/nix`, `/storage/bin/nix-shell`, `/storage/bin/nix-run`, `/storage/bin/nix-dev-shell`, and `/storage/bin/nix-portable` when they are detected as old nix-portable shims. Layer 4 owns unqualified `nix`, `nix-store`, and `nix-daemon` commands.

`nixctl install` also removes the known-bad manual `/storage/bin/nix-daemon` shim when it shells through `/storage/bin/nix daemon`.

## Recommendation

For Layer 4+ ROCKNIX systems, use real Nix binaries for remote store and profile work:

```sh
REAL_NIX=/nix/var/nix/profiles/default/bin/nix
REAL_NIX_STORE=/nix/var/nix/profiles/default/bin/nix-store
```

Avoid routing remote `nix copy` through nix-portable once the real `/nix` store is mounted and populated.
